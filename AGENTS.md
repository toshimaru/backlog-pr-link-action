# Repository Guidelines

## Project Structure & Module Organization

This repository contains a TypeScript GitHub Action that links a GitHub pull request to a Backlog issue.

- `src/main.ts` is the action entry point and GitHub Actions integration layer.
- `src/client.ts` contains Backlog API client behavior.
- `__tests__/` contains Jest tests, currently named with the `*.test.ts` pattern.
- `action.yml` defines the published action interface.
- `dist/` contains the bundled action output generated by `ncc`; update it when source changes affect runtime behavior.
- `deno.jsonc`, `tsconfig.json`, and `package.json` define formatting, linting, TypeScript, test, and build settings.

## Build, Test, and Development Commands

- `npm install` installs Node dependencies from `package-lock.json`.
- `npm test` runs Jest with `ts-jest` and ESM support.
- `npm run lint` runs Deno lint against `src`.
- `npm run fmt` formats TypeScript using `deno fmt`.
- `npm run build` bundles `src/main.ts` into `dist/index.js` with `@vercel/ncc`.
- `npm run release` runs `standard-version` for changelog and version updates.

Use Node.js `>=24`, as specified in `package.json`.

## Coding Style & Naming Conventions

Write TypeScript as ES modules and keep strict type checking clean. Formatting is managed by Deno: 2-space indentation, single quotes, no semicolons, LF line endings, and a 100-column line width. Prefer focused functions with explicit types where inference is unclear. Use descriptive camelCase names for variables and functions, and keep test files named `*.test.ts`.

## Testing Guidelines

Jest is the test framework. Add or update tests in `__tests__/` for behavior changes, especially Backlog URL parsing, API client behavior, and GitHub event handling. Run `npm test` before opening a pull request. When changing formatting or lint-sensitive code, also run `npm run lint` and `npm run fmt`.

## Commit & Pull Request Guidelines

Recent history uses Conventional Commit-style messages, such as `chore: update lockfile`, `build(deps): bump ...`, and `chore(release): 2.2.1`. Follow that style with concise scopes when useful.

Pull requests should include a clear description, the motivation for the change, linked issues when applicable, and notes about test coverage. For source changes that affect the action runtime, include the updated `dist/index.js` bundle in the same PR.

## Security & Configuration Tips

Do not commit Backlog API keys or GitHub secrets. Use repository or organization secrets such as `BACKLOG_API_KEY` in workflows. Keep action inputs documented in `action.yml` and reflected in README examples when behavior changes.

## Agent-Specific Instructions

Use the `gh` command line tool to interact with GitHub repositories.
